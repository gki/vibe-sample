# PRDレビュー

このドキュメントには、PRD（Product Requirement Document）のレビュー観点を記載します。

## レビューの目的

PRDレビューは、プロダクト要件の品質を確保し、開発チーム全体の共通理解を促進するために実施します。適切なレビューにより、以下の効果が期待できます：

- 要件の明確化と具体化
- 要件の漏れや矛盾の早期発見
- 実装可能性の確認
- チーム全体での要件理解の統一

## レビューの基準

### 共通PRDのレビュー観点

共通PRD（`prd-common.md`）については、以下の観点でレビューします：

- **対象ユーザーとペルソナの具体性と適切性**
  - ペルソナが具体的に定義されているか（名前、属性、行動パターン、ニーズなど）
  - ターゲットユーザー層が明確か
  - ペルソナがプロダクトの目的と整合しているか

- **用語集の網羅性と明確性**
  - プロジェクトで使用される主要な専門用語が網羅されているか
  - 各用語の定義が明確で理解しやすいか
  - 曖昧な用語が定義されているか

- **共通UI/UXルールの定義**
  - **通信中、通信エラー時の共通ルール**が定義されているか
    - 画面表示時通信（データ取得時）の表示方法が定義されているか
    - 画面操作時通信（データ送信時）の表示方法が定義されているか
    - 通信エラー時の表示方法とエラーハンドリングが定義されているか
  - **共通フィードバックUIルール**が定義されているか
    - エラーフィードバックの表示方法が定義されているか
    - ダイアログの使用方法と表示ルールが定義されているか
    - モーダルの使用方法と表示ルールが定義されているか

### 個別PRDのレビュー観点

個別PRD（各機能のPRD）については、以下の観点でレビューします：

- **機能概要の明確性**
  - 機能の目的が簡潔に説明されているか
  - 機能の価値が明確か

- **目的と背景の明確性**
  - 解決すべき課題が明確に記述されているか
  - ユーザーのニーズとの関連が明確か

- **機能要件の具体性と網羅性**
  - 機能の詳細仕様が具体的に記述されているか
  - **入力と出力の記載方法**: 入力と出力が、ユーザーが実際に画面で目にする内容や操作として記載されているか。APIのリクエスト/レスポンス形式やデータ構造が記載されていないか
    - 入力: ユーザーの操作や入力内容（ボタンクリック、入力フィールドへの文字入力など）として記載されているか
    - 出力: ユーザーが画面で目にする結果（表示の変化、状態の変化など）として記載されているか
  - 必要な機能が漏れなく記載されているか
  - UI/UX要件が具体的に記述されているか
  - データ要件が明確に定義されているか
  - **データ要件の記載方法**: 他の機能PRDのデータ要件に依存する場合は、詳細を記載せず参照のみになっているか
  - ビジネス上発生が想定されるエラーパターンが記載されているか
  - **実装詳細の抽象化**: 具体的な実装詳細（APIエンドポイント名、GraphQLクエリ名/ミューテーション名、クライアントライブラリ名、ファイルパスなど）が記載されていないか。データの保存・取得方法は抽象的な記述（例：「APIを通じて取得する」）に留められているか

- **制約事項と前提条件の明示**
  - **業務的な制約事項**が明示されているか
    - ビジネス上の制約（期間、リソース、業務ルールなど）が記載されているか
    - 業務的な制約事項が特にない場合、共通PRDに準拠する旨が記載されているか
  - **業務的な前提条件**が明確か
    - 外部サービスの利用、他システムとの連携、業務プロセス上の前提などが記載されているか
    - 業務的な前提条件が特にない場合、「前提条件は特にありません。」と記載されているか
  - **記載の適切性**: 技術的な制約や技術的な前提条件が記載されていないか

- **関連PRDとの整合性**
  - 共通PRDとの整合性が保たれているか
  - 他の機能PRDとの依存関係が明確か
  - 矛盾や重複がないか

- **共通UI/UXルールへの準拠**
  - **通信ハンドリング**が共通ルールに準拠しているか
    - 画面表示時の通信（データ取得時）の処理が共通ルールに準拠しているか
    - 画面操作時の通信（データ送信時）の処理が共通ルールに準拠しているか
    - 通信エラー時の処理が共通ルールに準拠しているか
  - **共通フィードバックUIルール**に準拠しているか
    - エラーフィードバックの表示が共通ルールに準拠しているか
    - ダイアログ、モーダルの使用が共通ルールに準拠しているか
  - **記載方法の適切性**: 共通ルールと完全に同じ場合は、1文で記載されているか。差異がある場合のみ、その差異が明記されているか
  - 共通ルールに準拠していない場合、その背景が合理的に説明されているか
    - 準拠しない理由が明確に記述されているか
    - 代替案や例外として適切に処理されているか

## 必須チェック項目

### 共通PRDの必須項目

- [ ] 対象ユーザーとペルソナが定義されているか
- [ ] 用語集が記載されているか（該当する場合）
- [ ] 共通UI/UXルール（通信中・通信エラー時）が定義されているか
- [ ] 共通フィードバックUIルール（エラーフィードバック、ダイアログ、モーダル）が定義されているか

### 個別PRDの必須項目

- [ ] 機能概要が記述されているか
- [ ] 目的と背景が記述されているか
- [ ] 機能要件が具体的に記述されているか
- [ ] 入力と出力が、ユーザーが実際に画面で目にする内容や操作として記載されているか（APIのリクエスト/レスポンス形式やデータ構造が記載されていないか）
- [ ] ビジネス上発生が想定されるエラーパターンが記載されているか
- [ ] 制約事項と前提条件が明示されているか
- [ ] 技術的な制約や技術的な前提条件が記載されていないか（業務的な制約事項・前提条件のみが記載されているか）
- [ ] データ要件の記載が適切か（他の機能PRDに依存する場合は参照のみになっているか）
- [ ] 関連するPRD（共通PRDを含む）への参照が記載されているか
- [ ] 共通UI/UXルールへの準拠状況が確認されているか（準拠しない場合は理由が記述されているか）
- [ ] 具体的な実装詳細（APIエンドポイント名、GraphQLクエリ名/ミューテーション名、クライアントライブラリ名、ファイルパスなど）が記載されていないか。データの保存・取得方法は抽象的な記述に留められているか
